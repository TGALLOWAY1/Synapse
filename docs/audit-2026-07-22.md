## Executive summary

Synapse remains a well-architected codebase — the safety gate is fail-closed and
enforced in code, the DAG pipeline and JSON-repair are careful, the write barrier
and planning authority model are genuinely sound, the sync revision guard is
atomic, and the image stores were correctly consolidated. The problems cluster in
five places, and they are consequential:

1. **Truncation is invisible end-to-end.** No generation call inspects Gemini's
   `finishReason`. A PRD section that hits its 8192-token ceiling is silently
   "repaired," marked complete, and used to build every downstream artifact; a
   core artifact that truncates is stored as raw broken JSON with slot status
   *done*. Both are silent corruption of generated artifacts with zero user
   signal (`geminiClient.ts:345`, `progressivePrdGeneration.ts:184`,
   `coreArtifactService.ts:742`).

2. **Deletion is broken across devices.** Soft-delete doesn't bump the revision
   and reconcile can't tell "server never had it" from "server deleted it," so a
   second signed-in device resurrects deleted projects everywhere. The sign-out
   namespace-wipe runs while sync is still live and is one swallowed logout error
   away from mass-deleting the user's cloud projects (`projectsStore.js:252`,
   `projectServerSync.ts:386`, `authStore.ts:56`).

3. **The default-on consistency review is structurally wasteful.** It must echo
   the entire merged PRD inside an 8192-token cap that the PRD routinely exceeds,
   so on large PRDs the pass truncates, fails its own detail guard, and is
   discarded — a guaranteed wasted Pro/Flash call on every rich generation
   (`prdConsistencyReview.ts:117`).

4. **The regression safety net you think you have doesn't exist.** There is no
   Playwright assertion suite, and `mockup-eval-harness.mjs` scores strings it
   writes itself against a mockup format the product retired — yet it gates PRs
   in CI. The happy path, conflict UI, and demo contract have zero end-to-end
   protection.

5. **A large dead subsystem still gates development.** The HTML-mockup pipeline is
   gone from the product but its CI job, eval harness, generated CSS, and docs
   remain, blocking PRs in the hottest area of the codebase.

No unconditional data-loss bug was found, and no reachable XSS sink exists today.
The truncation and deletion clusters are the ones to fix first.

## Architecture as-built

The three orientation passes matched the documented architecture closely; the
notable divergences are themselves findings (flagged **[drift]**).

- **SPA + serverless.** React 19 SPA (`src/`), Vercel Node serverless functions
  (`api/`, plain JS, not type-checked). Routing in `src/App.tsx` (static imports,
  `GlobalErrorBoundary`): `/`, `/tour`+`/about`, `/p/:projectId`, `/metrics`,
  `/developer/llm-trace`, `/privacy`, `/admin/recruiters`.
- **State.** One `useProjectStore` (`persist`) composing 12 slices; the heaviest
  are `downstreamUpdatePlanSlice.ts` (927 L), `spineSlice.ts` (884 L),
  `reviewSlice.ts`. Persistence is a bespoke 500 ms debounced, per-user-namespaced
  localStorage adapter with quota recovery + unload flush (`storage.ts`,
  post-PR #300). Transient job/progress state is stripped via `partialize`;
  `onRehydrateStorage` converts interrupted runs to settled errors.
- **Generation path.** `HomePage.handleCreateProject` → preflight →
  `runPrdGeneration.ts` → `prdService.generateStructuredPRD` →
  `progressivePrdPipeline` (section DAG, fast/strong tiers, concurrency caps) →
  `prdSectionMerge` → store via `updateSpineStructuredPRD`. Artifacts:
  `artifactJobController.ts` (1044 L) → `coreArtifactService` → `callGeminiStream`.
- **LLM transport.** `geminiClient.ts` is the single text chokepoint (watchdog
  timeout, network retry, stream retry loop). Images: browser →
  `/api/image/generate` proxy → `gpt-image-2`. **No zod** — every model-JSON
  boundary is hand-rolled guards + `as` casts; escape-hatch density at the parse
  boundary is genuinely low.
- **Sync.** `projectServerSync.ts` (1.5 s debounce) ↔ `api/projects.js` /
  `projectsStore.js`; monotonic `revision`, atomic guarded `updateOne`, 409 →
  explicit resolution. Images: IndexedDB source-of-truth + content-addressed
  Vercel Blob refs.
- **Auth.** Stateless HMAC-signed cookie (`session.js`), 30-day age check, no
  server-side session table consulted at verify. AES-256-GCM provider-key vault
  (`cryptoVault.js`), key bound to `userId:provider` via AAD.

**[drift] The documented and actual do not match in six places:**
- README.md:119 claims keys are "never returned to the browser as plain text,"
  but `GET /api/provider-keys?material=gemini` returns the decrypted Gemini key in
  plaintext JSON (`provider-keys.js:60-78`). The code comment admits it as a
  deliberate tradeoff; the public README states the opposite.
- README tech stack (README.md:95) omits the image model (`gpt-image-2`),
  contrary to the CLAUDE.md README rule requiring the image model be listed.
- CLAUDE.md states "The repo currently sits at 11" serverless functions; the
  actual count is **9**.
- `session.js:82-84` comments "Prefer the `__Host-` prefix in production"; the
  cookie name stays `synapse_session` — the prefix is never applied.
- `generationStages.ts:10` points readers at `prdPipeline.ts:phaseFor`, which
  does not exist; `mockup-eval-harness.mjs` claims lockstep with
  `buildMockupSrcDoc.ts`, which was deleted.
- The user-account collection is literally named `recruiters`
  (`users.js:5`), with `recruiter_sessions`/`recruiter_activity` and a live
  `/api/admin/recruiters` PII dashboard — a recruiter-portal pivot artifact.

**On your brief:** you described "a Playwright end-to-end suite with LLM-as-judge
quality scoring." That is not what exists. `scripts/e2e-live-run.mjs` is a
Playwright *screenshot-capture* harness with **zero assertions**; the "critique"
is an agent reading PNGs per `.claude/skills/e2e/SKILL.md`. The only automated
scoring is `scripts/mockup-eval-harness.mjs`, which is deterministic regex
heuristics — and it scores a retired format (see Test coverage). There is no
LLM-as-judge anywhere in the repo.

## Bugs

### P0 — silent corruption / broken core flow

**BUG-1 · Truncated PRD sections are silently "repaired," marked complete, and
build every downstream artifact.** _Confirmed._
`progressivePrdGeneration.ts:184-198` runs `repairTruncatedJson` and returns the
salvaged partial object with the `repaired` flag **discarded**; the DAG worker
(`:463-483`) then fires `section_completed` and turns the grid green. The root
cause is transport-level: `callGemini` (`geminiClient.ts:345-360`) only handles
`finishReason === 'SAFETY'` and empty text — a `MAX_TOKENS` finish returns the
truncated body as success. Sections are capped at `maxOutputTokens: 8192`
(`progressivePrdGeneration.ts:206`). `jsonRepair.ts:10-14` explicitly says callers
should treat repaired output as "use what you have, but warn the user"; its
primary caller warns no one.
*How it fails / what the user sees:* a rich `features` section that hits the cap
loses its tail features; the PRD shows no `failedSections`, no banner, and every
artifact (screens, data model, plan, mockups) generates from the shortened list.
Zero signal. This is the P0 "silent corruption of generated artifacts" case; the
trigger is real but conditional on a section exceeding ~8 K output tokens.
*Fix direction:* thread `finishReason`/`repaired` into section status +
`generationMeta.truncatedSections`, surface it like `failedSections`, and treat
`MAX_TOKENS` as a retryable section error.

**BUG-2 · Core artifacts have no output cap and store raw unparseable JSON as a
"done" artifact on truncation.** _Confirmed (per-subtype rendering:
needs-verification.)_
`generateCoreArtifact` calls `callGeminiStream` with no `maxOutputTokens`
(`coreArtifactService.ts:705-715, 751-761`) despite the client header warning that
the ~8 K default truncates rich structured output. On truncation `JSON.parse`
(`:719`) throws and the bare `catch` (`:742-747`) stores the **raw broken JSON**
as the artifact body — no `repairTruncatedJson`, no `finishReason` check, no
metadata flag, not even a `console` line. Blocking validation lets it through:
feature names appear inside the JSON text, so `screen_inventory`,
`implementation_plan`, and `design_system` pass all blockers and are saved with
slot status *done* (`artifactJobController.ts:402-405`). `design_system`
additionally loses its `tokens`/`tokensHash` (`:734-740`), so mockup prompts and
freshness anchoring silently degrade.
*What the user sees:* a green "done" artifact whose body is a wall of malformed
JSON; mockups generated without brand tokens.
*Fix direction:* set explicit per-subtype `maxOutputTokens`, consume
`onFinish.finishReason`, attempt repair before the raw fallback, and stamp a
blocker so the slot reads `needs_review` rather than `done`.

**BUG-3 · Cross-device deletion is broken: deleted projects resurrect on every
device; deletion is effectively impossible with a second device signed in.**
_Confirmed._
Two compounding defects. (a) `softDeleteProject` sets the tombstone with `$set`
only — **no `$inc: revision`** (`projectsStore.js:252-263`) — so a guarded push
from a device still holding the pre-delete baseline matches the tombstoned row and
its `$set` writes `deletedAt: null`, silently resurrecting it with no conflict.
(b) Reconcile computes `toPush = localIds − serverIds` and unconditionally
re-uploads (`projectServerSync.ts:386-392`), unable to distinguish "server never
had it" from "server soft-deleted it" even though `meta.lastSeenServerRevision`
proves the project reached the cloud. Delete on device A → device B's next
reconcile (sign-in, reload, or the `online` listener at `:570`) restores it to the
server → device A pulls it back as `toAdd`.
*What the user sees:* deleted projects reappear on all devices; this is the
"projects reappearing" class from `projects-disappearing-2026-06-27.md`,
mechanized. Broken core flow.
*Fix direction:* `$inc` revision on soft-delete; in reconcile, treat "absent from
server but has a recorded server baseline" as a remote delete, not a fresh
migration.

### P1 — user hits this in normal use, app misbehaves visibly

**BUG-4 · The default-on consistency review is structurally guaranteed to waste a
call on every large PRD.** _Confirmed._
`reviewPrdConsistency` requires the model to return the **entire corrected PRD**
under `maxOutputTokens: 8192` (`prdConsistencyReview.ts:107-121`), while the input
PRD is the merge of up to 8 sections each generated with an 8192 cap. On any PRD
whose JSON exceeds ~8 K tokens the reply truncates, `repairTruncatedJson` closes
it mid-array, and the 70 % detail-loss guard (`:130-149`) rejects the revision.
The pass runs by default and silently (`progressivePrdPipeline.ts:255-259`), so
the user never learns their reconciliation pass was discarded — and it costs
~8–12 K input + 8 K output Flash tokens (~15–25 % of a run) every time.
*Fix direction:* return a patch/field-edit list instead of the whole PRD, or gate
on serialized-PRD size and raise the cap; reject any *repaired* (truncated) review
outright rather than evaluating it.

**BUG-5 · A section retry can silently revert a decision the user confirmed while
it was running.** _Confirmed._
`handleRetrySection` invokes the write barrier with the spine id but **no
`expectedPrdHash`** (`ProjectWorkspace.tsx:859`), while the `decision_edit` amend
path mutates the latest spine's `structuredPRD` **in place under the same id**
(`spineSlice.ts:397-448`). The barrier's id check (`:560`) therefore passes even
though the content changed underneath. Sequence: confirm decision A → click "Run
again" on a failed section (snapshots the PRD) → confirm decision B during the
10–40 s call (amends in place) → retry completes, barrier passes the id-only
check, appends a PRD built from the pre-B snapshot. Decision B is reverted in the
working plan while its `PlanningRecord` still shows it confirmed. This is exactly
what `PLANNING_AND_DECISIONS.md` says the barrier prevents.
*Fix direction:* one line — pass
`expectedPrdHash: planningContentHash(activeSpine.structuredPRD)` (the barrier and
the decision-impact path already support it), or make the amend path
identity-changing.

**BUG-6 · Branch consolidation drops the persisted `safetyReview`, so the
restriction directive silently vanishes from all downstream artifacts.**
_Needs-verification (data path traced; runtime confirmation of a restricted
project recommended)._
`mergeBranch` builds the new "latest" `SpineVersion` from model output but copies
only `promptText` — not `structuredPRD` or `safetyReview` — and never
re-classifies (`branchSlice.ts:97-109`, `branchService.ts:10`). For a project the
classifier rated `allowed_with_restrictions`, a doc-wide "rewrite the entire PRD"
consolidation (`ConsolidationModal.tsx:47`) produces a canonical spine with
`safetyReview: undefined`, so the binding directive `canonicalPrdSpine.buildSafety`
(`canonicalPrdSpine.ts:244-270`) threads into every artifact disappears, and the
`SafetyBoundariesCard` vanishes. The parallel loss of `structuredPRD` also
degrades the spine to legacy read-only markdown.
*Fix direction:* carry `oldSpine.safetyReview` + `structuredPRD` onto the merged
version, or re-run `classifyProjectSafety` on doc-wide rewrites.

**BUG-7 · Sign-out mass-deletes cloud projects when a logout request fails.**
_Confirmed (latent; currently masked)._
`authStore.ts:56-67` calls `applyProjectUser()` **before** `stopProjectSync()`;
`projectUserSync.ts:104` does `setState(emptyPersistedState())`, which
synchronously fires `handleStoreChange` → `deleteRemote()` — an immediate,
non-debounced server soft-DELETE for every project (`projectServerSync.ts:441-451`)
— plus baseline erasure. This is harmless only because logout usually invalidates
the cookie first. But `recruiterApi.ts:133-139` never checks `resp.ok` and
swallows errors: a 500/proxy failure from `/api/auth/logout` leaves a valid cookie,
and the next lines mass-delete the user's entire cloud set (partially hidden by
BUG-3's resurrection on the origin device only).
*Fix direction:* `stopProjectSync()` before `applyProjectUser(null)`; drive remote
deletes only from the explicit delete action, not store-diff inference; check the
logout response.

**BUG-8 · Every mockup regeneration re-renders all screens through GPT-Image with
no prompt-hash reuse.** _Confirmed._
`runMockupSlot` fires `imageStore.generate` for every screen after each spec
version (`artifactJobController.ts:556-571`); records are keyed
`${versionId}:${screenId}:${quality}` and dedup only on in-flight state
(`mockupImageStore.ts:186-243`) — never on prompt identity. But the spec is
deterministic (`mockupService.ts:14-24`, no LLM) and `buildScreenImagePrompt` is
pure, so regenerating an unchanged mockup produces byte-identical prompts and
re-buys N brand-new image renders. Screen ids are fresh `uuidv4()` per derivation
(`mockupService.ts:214`), so there is no stable join to reuse across versions.
Every "Update mockup," every new spine version, every `retrySlot('mockup')` =
N images re-billed for pixels already generated. This is the single largest
per-action dollar cost.
*Fix direction:* hash prompt+size+quality, copy the prior record forward on match,
re-render only changed screens.

**BUG-9 · "Regenerate" silently discards the user's preflight clarification
answers.** _Confirmed._
`handleRegenerate` rebuilds from `latestSpine.promptText` (the raw idea) and never
reconstructs a `PreflightContext` from `latestSpine.preflightSession`
(`ProjectWorkspace.tsx:725-822`); `regenerateSpine` also doesn't carry
`preflightSession` forward (`spineSlice.ts:95-104`). The original run injected
those answers as "authoritative intent" (`prdService.ts:104-109`); the regenerated
PRD ignores them, drifting from the intent the user explicitly clarified, with no
warning. (This handler is also a third divergent inline copy of the flow
`runPrdGeneration.ts` is supposed to own — see Refactor.)
*Fix direction:* rebuild the clarification block from the persisted session and
route regenerate through `runPrdGeneration`.

**BUG-10 · Reconcile-on-signin overwrites local edits made during the fetch
window.** _Confirmed._
`projectServerSync.ts:311-341` decides `toRefresh` from list-time meta ("clean
local"), then awaits per-project `fetchBundle` before `applyBundlesOverwrite` +
`recordPulledBaseline` (`:367-380`) replace local slices and force
`hasUnsyncedChanges: false`. Edits made in that multi-second window — plausible
with the `online`-event reconcile (`:570`) firing while the user works after a
network flap — are overwritten with no conflict and their dirty flag cleared.
*Fix direction:* re-check the durable dirty flag per project at apply time; if
dirty, mark conflict instead of overwriting.

### P2 — edge case, degraded experience, or latent

- **BUG-11 · Section retry re-runs with the identical 8192 cap that truncated the
  original** (`prdSectionRetry.ts:115-123`; same in `groundingService.ts:41-47`).
  For a deterministically over-length section every retry is a near-guaranteed
  repeat failure and full re-spend. _Confirmed._ *Fix:* on `MAX_TOKENS`, raise the
  cap or add a "emit fewer items" directive.
- **BUG-12 · Artifact prompts ship the canonical spine AND the full PRD markdown
  appendix, ×7 artifacts, uncapped** (`coreArtifactService.ts:609-621`,
  `buildArtifactPrompt`). The spine is derived from the PRD, so most content is
  sent twice per call — ~40–70 K redundant input tokens per bundle. _Confirmed._
  *Fix:* drop/truncate the appendix when `spineContextUsed` is true.
- **BUG-13 · Artifact and image runs report $0 in the Metrics dashboard**
  (`artifactJobController.ts:585-586` never sets tokens; `generateCoreArtifact`
  never passes `onUsage`, though `callGeminiStream` now captures it at
  `geminiClient.ts:503-510`). The most expensive actions read as free.
  _Confirmed._ *Fix:* ~10 lines to thread `onUsage` into `nodeObs`.
- **BUG-14 · Device-B image hydration false-negative.** `loadForVersion` stamps
  `loadedVersions[versionId] = true` even when refs haven't arrived yet
  (`mockupImageStore.ts:144-162`); `setProjectRefs` never invalidates it, and refs
  are pulled fire-and-forget after reconcile (`projectServerSync.ts:405-410`).
  First visit to a project on a new device shows missing mockups and the presence
  UI confidently claims none exist until a remount. _Confirmed._ *Fix:* invalidate
  `loadedVersions` on `setProjectRefs`; refresh refs on project open.
- **BUG-15 · No AbortController on the PRD path — generation cannot be
  cancelled.** The transport honors `AbortSignal` and `generateStructuredPRD`
  accepts `options.signal` (`prdService.ts:91,120`), but `runPrdGeneration.ts`,
  `HomePage.tsx:217`, `PreflightView.tsx:121`, and `handleRegenerate`
  (`ProjectWorkspace.tsx:743`) never create one, and there is no cancel affordance
  in `components/progress/`. Navigating away burns all section calls + the review
  to completion. _Confirmed._ *Fix:* create a per-run controller, thread as
  `options.signal`, abort on unmount/project switch, add a cancel button.
- **BUG-16 · Readiness checkpoints go stale during live artifact generation.**
  `buildReadinessReviewInputFromState` folds live job slot status into the
  readiness snapshot hash (`readinessSlice.ts:64-69` → `outputAlignment.ts:267` →
  `readinessReview.ts:393-408`); each of the 7 concurrent slots settling changes
  the hash, so a checkpoint created mid-generation is rejected `'stale'` at
  commit. Not a deadlock (escape hatches verified), but avoidable churn.
  _Confirmed._ *Fix:* exclude transient `slotStatus` from the hash.
- **BUG-17 · Full-bundle regeneration on any PRD change.** `isSlotDoneForSpine`
  requires an exact `spineVersionId` match (`artifactJobController.ts:157-174`), so
  any PRD edit appends a new spine version and marks all 7 artifacts pending;
  `startAll` regenerates everything (+ images per BUG-8) even for a wording tweak
  in an unrelated section. `evaluateProjectFreshness` computes precise change-aware
  staleness (`artifactFreshness.ts:196`) but nothing on the generation path
  consults it. _Confirmed mechanism / magnitude needs-verification._ *Fix:* stamp
  an input-content hash per artifact version, skip on match (leave the advisory
  freshness layer untouched).
- **BUG-18 · `recordCloudSaved` clears the dirty flag even when edits arrived
  after the pushed snapshot** (`projectServerSync.ts:150,174`). If the tab closes
  before the next 1.5 s timer, durable meta claims "synced," reconcile won't
  retry, and the cloud stays stale on other devices until the next local edit.
  _Confirmed._ *Fix:* only clear if nothing changed since the snapshot.
- **BUG-19 · Logout does not revoke the session server-side.** The signed cookie
  is self-contained; `verifySessionToken` checks only signature + 30-day age
  (`session.js:53-78`), and `handleLogout` merely clears the cookie
  (`auth/email.js:139-143`). A captured token stays valid 30 days; a password
  change cannot invalidate existing sessions; there is no "log out other devices."
  _Confirmed._ *Fix:* add a per-user token epoch/`sessionVersion` checked at verify,
  bumped on logout-all and password change.

## Refactor & consolidation

Ranked by payoff/effort. Effort S/M/L.

1. **Delete the dead HTML-mockup pipeline — including the CI gate. (S, highest
   payoff.)** The product renders mockups exclusively via GPT-Image + SVG;
   `buildMockupSrcDoc.ts` / `MockupHtmlPreview` no longer exist, `mockupService.ts`
   emits only `mockup_spec_v1`, and no iframe exists in `src/`. Still present:
   `.github/workflows/mockup-harness.yml` — a **PR-blocking "Harness gate"** that
   triggers on `src/lib/mockup*.ts` (matches all four hot `mockupVariant*.ts`
   files) and type-checks with `npx tsc --noEmit` (the false-clean trap CLAUDE.md
   documents); `scripts/mockup-eval-harness.mjs` (606 L, comment claims lockstep
   with a deleted file); `scripts/build-mockup-css.mjs`,
   `scripts/mockup-tailwind.config.cjs`, `src/styles/mockup-tailwind.*`
   (`generated.ts` has zero importers); `harness/` (208 K); four npm scripts;
   `docs/mockup-evaluation-harness.md`. *Buys:* removes a PR gate that can fail
   active Screens/variant work for irrelevant reasons, deletes ~1,000+ lines,
   stops the docs describing a non-existent iframe. *Risk:* none — nothing live
   consumes it. Confirm the workflow is enabled in the Actions UI (deletion is
   warranted either way). Verify `MOCKUP_HTML_V1` (the one live remnant,
   `mockupParsing.ts:62`) isn't present in the pinned demo snapshot before dropping
   the parse branch — that's server data.

2. **Consolidate the FNV-1a `stableHash`/`canonicalStringify` — implemented 7×.
   (S–M.)** `designTokens/hash.ts:29`, `mockupVariantTrust.ts:67`,
   `screenReviewWorkflow.ts:592`, `review/hash.ts:11`, `planning/planningHash.ts:11`,
   `planning/assumptionImport.ts:31`, `dataModel/dataModelNavigation.ts:7`. These
   hashes drive staleness/trust/review-signature comparisons; a canonicalization
   fix applied to one copy silently changes hashes in one subsystem only, producing
   phantom "stale"/"changed" flags. *Buys:* one `src/lib/hash.ts`, ~150 lines gone,

## Test coverage gaps

**The eval harness measures nothing about the product.** `mockup-eval-harness.mjs`
`generateFixture` (`:172-209`) is a hard-coded template-string generator that
imports nothing from `src/` and calls no model — every score (quality weights
`:122-149`, thresholds quality<75/consistency<85 `:507-510`, perturbation ≥0.9
`:492`) measures strings the harness wrote three lines earlier, and the
"non-determinism" it detects is a literal `runIndex % 2 ? 'A' : 'B'` suffix it
planted. If the live product rendered identical generic output on every screen,
every metric stays green because the live path never runs. It also validates
`mockup_html_v1`, a format the product retired for `mockup_spec_v1` + GPT-Image.
**Verdict: it manufactures a false sense of evaluated quality and should be retired
or rewired to score real `mockup_spec_v1` payloads.** `e2e-live-run.mjs` is honest
about being a screenshot tool with zero assertions — real QA signal, but non-gating
and dependent on the reviewing agent.

**No Playwright assertion suite exists.** Top journeys with zero end-to-end
protection: (1) the entire happy path (create → preflight → generate → readiness
gate → commit → artifact bundle → export); (2) the cross-device conflict-resolution
UI (the store logic is well tested in `projectServerSync.test.ts`, but nothing
renders the banner/CTA the user operates); (3) the public demo contract (SYN-006,
still open); (4) storage-quota exhaustion → recovery; (5) interrupted-generation
reload → recovery → resume.

**Which findings the current suite would have caught:** essentially none of the P0/P1
bugs. BUG-3/BUG-7 (delete/sign-out) — the store *logic* is tested but the
delete-propagation and sign-out ordering are not; `projectServerSync.test.ts`
covers push/pull/conflict but not the `toPush = localIds − serverIds` deletion
inference. BUG-1/BUG-2/BUG-4 (truncation) — invisible: no test drives a
`MAX_TOKENS` finish, and the harness that should catch mockup regressions is
inert. BUG-5 (retry vs decision) — no test exercises a concurrent amend. BUG-8
(image dedup) — untested. **`runPrdGeneration.ts` has no test file at all** — the
glue that persists partials, safety reviews, and the error taxonomy onto the spine
is unprotected. `artifactJobController.ts` (1044 L) has two narrow tests; its
`executeJob` ordering, semaphore, `retrySlot` budget, mockup fan-out, and abort
path are untested. `progressivePrdPipeline.test.ts` has exactly one test.

**The prompt-snapshot net has real blind spots.** `promptSurfaces.test.ts` does not
cover prompts built outside `src/lib/prompts/` (`branchService.ts` consolidation +
`replyInBranch`, `prdConsistencyReview.ts` `SYSTEM`, `enhancePrompt`, the review
repair block), snapshots only artifact `system`+`userPrefix` (not the runtime
assembly), and — because it uses a tiny fixture — cannot see changes to *what* gets
serialized at runtime or to generation config (model routing, temperature,
`maxOutputTokens` — the actual cause of BUG-4).

## Feature recommendations

Ranked by leverage (user value ÷ cost), grounded in what exists.

1. **Truncation-aware generation. (Low cost, high value.)** Inspect
   `finishReason`/`onFinish` (already exposed at `geminiClient.ts:74`), set
   per-subtype `maxOutputTokens`, and surface a "section truncated — regenerate"
   affordance. Builds directly on existing `repairTruncatedJson`, `failedSections`,
   and section-status plumbing. *Fixes BUG-1/2/11 at the root and turns the biggest
   silent-corruption class into a visible, recoverable state.* Main risk: choosing
   caps that fit the model's real ceiling.

2. **Input-hash artifact & image caching ("regenerate only what changed"). (Medium,
   high value + direct cost reduction.)** Stamp a content hash (canonical-spine
   slice + dependency versions + image prompt) on each artifact/image version and
   skip regeneration on match. Builds on the already-computed
   `evaluateProjectFreshness` (`artifactFreshness.ts:196`, currently advisory-only)
   and the content-addressed image store. *Fixes BUG-8/17; cuts the dominant dollar
   and token cost.* Risk: must not disturb the advisory freshness layer (cross-cutting
   rule 9/10) — add a parallel generation-gate hash, don't repurpose the display one.

3. **Patch/diff-based consistency review and consolidation. (Medium, high value.)**
   Have the review and doc-wide consolidation return field-level edits instead of the
   whole PRD. *Fixes BUG-4 (structural waste) and the O(n²) consolidation cost
   (A1.2), and makes the review actually apply on the large PRDs that most need it.*
   Builds on the existing identity-based guards. Risk: the diff format needs its own
   validation, but the guard infrastructure is already there.

4. **A working cost dashboard. (Low, high visibility.)** Thread `onUsage` through
   `generateCoreArtifact` into `nodeObs` (~10 lines; the capture already exists at
   `geminiClient.ts:503`) and add per-image GPT-Image pricing. *Fixes BUG-13.* For a
   product whose pitch is disciplined planning, showing real per-run cost is
   on-brand and nearly free. It's also the prerequisite for any "budget guardrail"
   feature.

5. **Cancel / pause generation. (Low.)** Wire an `AbortController` through the PRD
   path (the signal plumbing already exists end-to-end) and add a cancel button to
   `components/progress/`. *Fixes BUG-15;* removes wasted spend on abandoned runs.

6. **A real regression suite to replace the inert harness. (Medium.)** A small
   Playwright assertion suite over the happy path + conflict UI + demo contract
   (SYN-006), plus a `MAX_TOKENS` fixture test for the pipeline. This is the
   capability you believed you already had; the screenshot harness and dev-skip-auth
   bypass are reusable scaffolding.

**Explicitly expensive given this architecture:**
- **Real-time multi-user collaboration.** The sync model is single-owner,
  last-writer-wins with a monotonic revision and manual conflict resolution
  (`projectsStore.js`, `projectServerSync.ts`). Concurrent multi-user editing would
  require OT/CRDT and a server authority rewrite — disproportionately expensive.
- **Additional model providers (the README roadmap lists Anthropic/Azure).** Model
  *tier routing* is clean (`artifactModelSettings.ts`), but the JSON boundary is
  Gemini-specific: `responseSchema`/`responseMimeType` and the browser-direct key
  path are wired through every parse site with no zod abstraction. A second provider
  touches every one of the 41 parse boundaries. Introducing a provider-agnostic
  response-validation seam (zod at the boundary) first would de-risk this
  substantially — and would also have caught BUG-1/BUG-2.
